# План усиления `DocModuleLoader`

## Цель
Повысить надежность динамической загрузки библиотек документов и диагностируемость ошибок.

## Проблемы сейчас
- Некорректные пути/отсутствие DLL → общий null результат, минимум контекста.
- Нет проверки совместимости версий/ABI между загружаемой DLL и хостом.
- Обработка baseDirectory была связана с CWD (частично исправлено).

## Доработки
1) Проверка пути и версии
- Перед загрузкой проверять существование DLL; логировать размер/время изменения.
- Считать `AssemblyName.GetAssemblyName(path)` и логировать версию.
- Сверять ожидаемую версию интерфейса (через общую константу/атрибут `RequiredDocApiVersion`).

2) Безопасная загрузка
- Ловить `BadImageFormatException`, `FileLoadException`, `ReflectionTypeLoadException` и логировать детали (loader exceptions).
- (Опционально) Таймаут/отмена загрузки при внешних зависимостях.

3) Создание экземпляра
- Проверять наличие публичного конструктора с нужной сигнатурой; при отсутствии — логировать понятную ошибку.
- Если типов‑наследников несколько — выбирать по атрибуту/интерфейсу‑маркеру.

4) Диагностика
- Расширенные логи: путь к DLL (абсолютный), версия, имя типа, длительность загрузки.
- В Development — подробный стек; в Production — краткий с correlation id.

5) Тесты
- Юнит‑тесты: отсутствующая DLL, несовместимая DLL, корректная DLL, отсутствие конструктора.
- Интеграционный: загрузка реального модуля и вызов метода.

## Этапы
- Этап 1: Валидации/логи, обработка исключений.
- Этап 2: Версионирование API, выбор типа по атрибуту.
- Этап 3: Тестирование (юнит/интеграция).

## Критерии готовности
- Детальные логи по всем ошибкам загрузки.
- Покрытие веток ошибок юнит‑тестами (≈100%).
