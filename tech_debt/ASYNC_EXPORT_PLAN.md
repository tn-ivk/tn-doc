# План асинхронного экспорта отчётов

## Цель
Убрать долгие операции экспорта из HTTP‑запросов, повысить надёжность и масштабируемость.

## Подход
- Команда экспорта кладётся в очередь (in‑memory/Redis/RabbitMQ).
- Воркер выполняет рендер/экспорт и складывает результат в хранилище (файловая система/объектное).
- Клиент получает `jobId` и опрашивает статус/скачивает по готовности.

## Доработки
1) Контроллер
- `POST /export` → `202 Accepted` + `jobId`.
- `GET /export/{jobId}/status` → состояние (queued/running/succeeded/failed).
- `GET /export/{jobId}/file` → отдача файла.

2) Очередь/воркер
- Интерфейс `IExportQueue`; реализация (сначала in‑memory, затем Redis/RabbitMQ).
- Бэкоф/повторы, дед‑letter на фейлах.

3) Хранилище файлов
- Папка экспорта с ограничением размера и сроком хранения; очистка по расписанию.

4) Тесты
- Юнит‑тесты очереди, статусов; интеграционные e2e на in‑memory.

## Критерии готовности
- Экспорт больших документов не блокирует HTTP‑потоки; устойчивые повторы.
