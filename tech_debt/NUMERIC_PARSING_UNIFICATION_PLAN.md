# План унификации парсинга чисел в `DocGeneral`

## Цель
Единообразный, безопасный парсинг числовых значений из строк с поддержкой знака и `,`/`.` в качестве разделителя.

## Текущее состояние
- В `DocGeneral` реализованы `StringToFloat`/`StringToDouble` с нормализацией и `InvariantCulture`.
- В коде встречались локальные `TryParse` и `Replace(',', '.')` (часть устранено).

## Доработки
1) Единая точка входа
- Использовать только `StringToFloat`/`StringToDouble` во всех документах.
- Для валидации — `CheckLegal` (или возврат `NaN` через fallback, если требуется именно проверка).

2) API
- Убедиться, что `StringToFloat/Double`:
  - Делают `Trim`, заменяют `,`→`.`.
  - `NumberStyles.Float` + `InvariantCulture`.
  - Поддерживают параметр `fallback` (по умолчанию 0; для проверок — `float.NaN`/`double.NaN`).

3) Ревизия мест использования
- Найти и заменить остатки локальных `TryParse` и ручных замен — на вызовы `DocGeneral`.
- Исключение: валидаторы UI/форматирование строк (оставить целевое форматирование).

4) Тесты
- Набор строк: `"-5"`, `"23.5"`, `"23,5"`, `"  12  "`, `""`, `null` — ожидания для float/double.
- Проверить `fallback` и ветки `NaN`.

## Риски
- Неожиданные места, где ожидалась запятая в выводе: это форматирование, не парсинг — не менять.

## Критерии готовности
- Все числовые парсинги проходят через `DocGeneral`.
- Юнит‑тесты зелёные.
