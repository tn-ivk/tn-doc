# План Security Hardening для TN_Doc

## Цели
- Минимизировать риск XSS, CSRF, Path Traversal, RCE через шаблоны/конфиги.
- Усилить аутентификацию/авторизацию, безопасные заголовки и политику контента.

## Доработки
1) Анти‑подделка запросов (CSRF)
- Включить анти‑подделку для всех state‑changing эндпоинтов (POST/PUT/DELETE) в MVC/razor формы.
- Проверка токена в фильтре/атрибутах.

2) Валидация входных данных
- В контроллерах: модел-валидация [ApiController]/ModelState.
- Для JSON: размер тела/DTO правила; ручные проверки для критичных полей (пути к файлам, имена шаблонов).
- Нормализация путей: только `AppContext.BaseDirectory.CombineSafe(relPath)`.

3) Безопасные заголовки
- Добавить middleware для `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `X-XSS-Protection: 0`.
- CSP (минимум): `default-src 'self'` (с учётом FastReport UI).
- HSTS в Production.

4) Аутентификация/Авторизация
- Ограничить доступ к экспорту/редактированию документов ролями.
- Централизованный `Authorize` на контроллеры/действия; проверка прав на IdDevice/IdDoc.

5) Журналы безопасности
- Логировать попытки доступа без прав, ошибки валидации входных данных, изменения словарей/конфигов.

6) Обработка файлов
- Проверять существование и тип файлов шаблонов; запрет `..`/абсолютных чужих путей.
- Ограничить размер экспортируемых файлов и время генерации.

7) Тесты
- Набор негативных тестов (подделки токенов, инъекции пути, слишком длинные поля, нет прав).

## Этапы
- Этап 1: заголовки, CSRF, CombineSafe и проверки путей.
- Этап 2: роли/права на действия; аудит логов.
- Этап 3: негативные тесты.

## Критерии готовности
- Негативные тесты проходят, заголовки выставляются, пути домаршрутизируются безопасно.
