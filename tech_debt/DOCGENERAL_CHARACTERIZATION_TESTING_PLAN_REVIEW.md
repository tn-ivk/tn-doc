## 1. Неточность: `BaseDocumentTest<T>` не «неиспользуемый»

**Раздел 1.3**, строка 38: *«BaseDocumentTest\<T\> присутствуют, но фактически не используются массово»*

**Факт:** Класс `BaseDocumentTest<T>` (`Tests.Shared/BaseTestClass.cs:239`) содержит готовые методы `AssertConstructorInitializesCorrectly()`, `AssertValidJson()`, `AssertValidHtml()`, `AssertFileExists()`. Это не мёртвый код — это подготовленная, но недоиспользованная инфраструктура.

**Предложение:** Переформулировать на «`BaseDocumentTest<T>` реализован, но не наследуется ни одним тестом — его следует активно использовать как базу для characterization-тестов модулей». Это устраняет ложное впечатление, что класс нужно переписывать, и направляет на reuse.

## 2. Полностью пропущена существующая инфраструктура `TestDataFixture`

**Нигде в плане** не упоминается `Tests/Tests.Shared/Fixtures/TestDataFixture.cs`, который уже содержит генераторы тестовых данных:

- `CreatePassportJson()`, `CreateActJson()`, `CreateKmhJson()`, `CreatePoverkaJson()`, `CreateReportJson()`, `CreateJornalJson()`
- `CreateMinimalEditFormHtml()`
- `GetKmhMeasurementData()` для разных типов КМХ

**Влияние:** Фаза 0 и задачи `TDOC-CHAR-001`–`TDOC-CHAR-005` планируют создавать инфраструктуру «с нуля», хотя значительная часть уже существует. Это может привести к дублированию или, хуже, к отказу от рабочего кода в пользу «нового».

**Предложение:** Добавить раздел «1.4a Существующая инфраструктура fixtures» с описанием `TestDataFixture` и `MockConfigHelper`. Задачи Фазы 0 скорректировать: не «реализовать с нуля», а «расширить существующий TestDataFixture snapshot-вариантами (raw + normalized)».

## 3. Неточное количество модулей: 41 → 44

**Раздел 1.4**, строка 51: *«41 doc type в CfgApp.json»*

**Факт:** Фактически в `tn.docgeneral/` присутствует **44 модульных директории** (без служебных `TN.DocGeneral`, `tn.utils`). Кроме того, план перечисляет документы по `CfgApp.json`, но реальное число модулей определяется количеством DLL-проектов, а не записей конфига — один тип документа может обслуживать несколько `IdDevice`.

**Предложение:** Уточнить формулировку: «44 модуля документных DLL × 3 устройства = до 132 уникальных комбинаций `(IdDoc, IdDevice)`». Это точнее оценивает объём матрицы тестирования.

## 4. Модули `ActProducer` и `ActRoute` отсутствуют в матрице приоритизации

## 5. Нет стратегии тестирования `SaveDoc` с rollback

**Раздел 4.3** упоминает *«safe-сценарии для SaveDoc (rollback/test records)»*, но без конкретной стратегии.

**Факт:** `DocPassport.SaveDoc()` выполняет `Database.ExecuteSqlRaw("UPDATE ...")` и `SaveChanges()` — и это прямые записи без транзакции. `DocReport.SaveDoc()` также использует `ExecuteSqlRaw`. Без явной стратегии изоляции тесты могут повредить данные.

**Предложение:** Добавить отдельный подраздел «3.6 Стратегия безопасного тестирования SaveDoc» с конкретными вариантами:

- **Вариант A:** Обёртка в транзакцию с `ROLLBACK` после каждого теста (требует, чтобы тестовый DbContext использовал `BeginTransaction`).
- **Вариант B:** Выделенные тестовые записи с фиксированными `id` (> 900000), удаляемые в teardown.
- **Вариант C:** Snapshot/restore БД перед/после каждого suite.

Указать, что для модулей с `ExecuteSqlRaw` (Report, Passport) варианты A ограничены, так как прямой SQL может не участвовать в EF-транзакции.

## 6. Недооценка сложности нормализации HTML

**Раздел 3.2** описывает нормализацию HTML как «trim/whitespace normalization, сортировка атрибутов, удаление шумовых значений».

**Факт:** `GetEditDoc()` в Passport генерирует HTML с:

- GUID-based `id` для элементов с пустым `Key` (`AdditionalInfoExtensions.GetId()` → `Guid.NewGuid()`)
- Динамическими `<select>` со значениями из конфига
- Встроенными `onchange` JavaScript-обработчиками с литеральными строками
- Условным ELIS-рендерингом (целые колонки могут присутствовать или нет)

Простого «trim + sort attrs» недостаточно — нужен целый pipeline обработки.

**Предложение:** Расширить задачу `TDOC-CHAR-003` (HTML normalizer). Описать минимальный набор трансформаций:

1. Замена GUID-подобных id на стабильные плейсхолдеры (`id="guid-*"` → `id="STABLE_ID_N"`)
2. Нормализация inline JS (или полное удаление `onchange`/`onclick`)
3. Канонизация пустых атрибутов и self-closing тегов
4. Удаление HTML-комментариев и пустых строк

Оценку трудозатрат для `TDOC-CHAR-003` увеличить с ~0.5 до ~1.5 дня.

## 7. CI-фильтры: проблема глубже, чем описано

**Раздел 1.3**, строки 41-44 верно указывает на несоответствие, но не раскрывает масштаб.

**Факт:**

- Unit фильтр `Tests.Controllers` **работает** с `Tests.Unit.controllers` (NUnit `Namespace~` — case-insensitive substring match)
- Integration фильтр `Tests.Libraries.*` **не находит ни одного теста** — namespace `Tests.Libraries` вообще не существует. Все интеграционные тесты в `Tests.Integration.*`

**Влияние:** Интеграционные тесты **вероятно не запускаются в CI вообще**. Это критическая проблема, не просто «несоответствие».

**Предложение:** Поднять приоритет задачи `TDOC-CHAR-302` из Эпика D в Фазу 0 (подготовка). Пока CI не запускает интеграционные тесты, добавлять новые characterization-тесты бессмысленно — они не будут валидироваться. Добавить конкретный fix: заменить `Tests.Libraries.*` на `Tests.Integration.*`.

## 8. Нет упоминания `CachedDocModuleLoader` как тестовой зависимости

**Факт:** `CachedDocModuleLoader` (`CachedDocModuleLoader.cs`) — LRU-кэш на 5 модулей с `ConcurrentDictionary`. Он загружает DLL через рефлексию и создаёт экземпляры `DocGeneral` через `Assembly.CreateInstance()`.

**Влияние на тестирование:** Characterization-тесты через `HomeController` неизбежно проходят через этот loader. Порядок запуска тестов может влиять на результат: если кэш заполнен 5 модулями, загрузка 6-го вытесняет один из предыдущих. Это потенциальный источник flaky.

**Предложение:** Добавить в раздел 7 (Риски) новый пункт «7.5 Риск: LRU-кэш модулей». Меры:

- Очищать кэш (`ClearCache()`) перед каждым test suite
- Либо тестировать модули напрямую (создавая экземпляры минуя loader), что устраняет зависимость от кэша

## 9. Отсутствие критерия выбора между JSON snapshot и semantic assertions

**Раздел 3.5** рекомендует «semantic checks поверх snapshot», но не даёт критерия, когда достаточно только snapshot, а когда нужны оба подхода.

**Предложение:** Добавить таблицу решений:

| Условие | Snapshot | Semantic | Оба |
|---------|----------|----------|-----|
| Стабильный модуль, редкие изменения | + | - | - |
| Модуль с ELIS-интеграцией | + | + | + |
| Модуль с DataARM/прямым SQL | + | + | + |
| Модуль с GUID в output | - | + | - |
| Новый/активно разрабатываемый модуль | - | + | - |

Это позволит разработчикам принимать решение по каждому модулю, а не гадать.

## 10. Нет упоминания кросс-платформенной специфики

**Факт:** Проект работает на Windows и Linux. `GetPathTemplateFile()` возвращает пути к `.frx`-шаблонам. `General.cs` использует `Path.Combine()`, но не везде. HTML-шаблоны (`DocEdit.html`) читаются напрямую.

**Влияние:** Characterization-тесты, снятые на Windows, могут давать другие результаты на Linux (разделители путей в данных, line endings в HTML, поведение `DESCRIBE` на разных версиях MySQL/MariaDB).

**Предложение:** Добавить в раздел 3.4 (нормализация) пункт о кросс-платформенных различиях:

- Нормализация line endings (`\r\n` → `\n`)
- Нормализация путей в output (`\\` → `/`)
- Указать, что baseline снимается на одной платформе и фиксируется (например, Linux для CI)

## 11. Roadmap не учитывает зависимость Фазы 1 от исправления CI

**Раздел 8.2**, Неделя 1 параллельно планирует «пилот 2-3 модуля» и «стабилизировать diff-отчёты», но не включает исправление CI-фильтров.

**Влияние:** Если пилотные characterization-тесты попадут в `Tests.Integration`, они не будут запускаться в CI (см. замечание 7). Три стабильных прогона (критерий выхода Фазы 1) будут проверены только локально.

**Предложение:** Перенести `TDOC-CHAR-302` (исправление CI-фильтрации) в начало Недели 1, перед пилотом. Последовательность: fix CI → пилот → стабилизация.
