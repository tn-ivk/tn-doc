## R5-1. Неявные межэпические зависимости не декларированы в backlog

**Критичность: Средняя**

**Раздел 9** (Backlog) перечисляет 4 эпика (A–D), но не содержит явных зависимостей между задачами разных эпиков. Зависимости подразумеваются через фазовую модель (раздел 4: Ф0 → Ф1 → Ф2 → Ф3), но при конвертации backlog в трекер задач без формальных `blockedBy` зависимости могут быть потеряны.

**Факт:** Логический анализ задач выявляет следующие неявные зависимости:

| Блокирующая задача | Блокируемые задачи | Обоснование |
|---|---|---|
| `TDOC-CHAR-000` (аудит) | `TDOC-CHAR-001`–`TDOC-CHAR-007` | Аудит определяет формат, инструменты и стратегии для остальных инфраструктурных задач |
| `TDOC-CHAR-002` (JSON normalizer) | `TDOC-CHAR-101`–`TDOC-CHAR-103` (пилот) | Пилот невозможен без normalizer для сравнения baseline |
| `TDOC-CHAR-003` (HTML normalizer) | `TDOC-CHAR-101`–`TDOC-CHAR-103` (пилот) | Аналогично: `GetEditDoc` возвращает HTML |
| `TDOC-CHAR-004` (snapshot comparator) | `TDOC-CHAR-101`–`TDOC-CHAR-103` (пилот) | Без comparator нет golden-master |
| `TDOC-CHAR-006` (стратегия SaveDoc) | `TDOC-CHAR-101`, `TDOC-CHAR-102` | Passport и Report имеют write-path, требующий утверждённой стратегии изоляции |
| `TDOC-CHAR-007` (preflight check) | `TDOC-CHAR-101`–`TDOC-CHAR-103` (пилот) | Без preflight тесты будут падать на `FileNotFoundException` |
| `TDOC-CHAR-302A` (CI hotfix) | `TDOC-CHAR-101`–`TDOC-CHAR-105` (пилот) | Уже декларировано в Эпике D |

**Влияние:**
- При импорте задач в Jira/GitLab/Azure DevOps без `blockedBy` связей команда может начать пилот до готовности инфраструктуры.
- Фазовая модель (раздел 4) подразумевает эту последовательность, но backlog (раздел 9) — самодостаточный список задач, который может использоваться автономно.

**Предложение:** Добавить в начало раздела 9 или в каждый эпик явную секцию зависимостей:
```
### Межэпические зависимости
- Эпик B (пилот) блокируется: TDOC-CHAR-002, 003, 004, 006, 007, 302A
- Эпик C (масштабирование) блокируется: TDOC-CHAR-104, 105 (стабилизация пилота)
- TDOC-CHAR-302B блокируется: TDOC-CHAR-301 (уже декларировано)
```

## R5-2. Roadmap «4 недели» несогласован с суммой фаз

**Критичность: Средняя**

**Раздел 8.1** (строки 561–564) оценивает фазы:

| Фаза | Оценка (дни) |
|------|-------------|
| Ф0 | 4–6 |
| Ф1 | 5–7 |
| Ф2 | 7–10 |
| Ф3 | 5–8 |
| **Сумма** | **21–31** |

**Раздел 8** (строки 568–569) утверждает:
- «минимальный результат: **2 недели** (Ф0 + Ф1 + часть Ф2)»
- «расширенный результат: **4 недели** (Ф0-Ф3)»

**Несогласованность:**

1. **Минимальный (2 недели = 10 рабочих дней):** Ф0(4) + Ф1(5) = 9 дней + «часть Ф2» — при минимальных оценках укладывается в 2 недели. Приемлемо.

2. **Расширенный (4 недели = 20 рабочих дней):** Ф0(4–6) + Ф1(5–7) + Ф2(7–10) + Ф3(5–8) = 21–31 день. Даже при минимальных оценках (21 день) это **4.2 недели**, а при средних (~26 дней) — **5+ недель**. Утверждение «4 недели» достижимо только при параллелизации фаз или при нижней границе всех оценок.

**Влияние:**
- При планировании спринтов команда может заложить 4 недели, а реально потребуется 5–6.
- Краткое резюме (строка 14) дублирует эту оценку: «расширенный: 4 недели».

**Предложение:** Один из вариантов:
1. Скорректировать на «расширенный результат: 4–6 недель (Ф0-Ф3)».
2. Указать, что 4 недели достижимы при параллелизации (Ф2 частично перекрывается с Ф1, Ф3 частично с Ф2).
3. Уточнить, что «4 недели» — это оптимистичная нижняя граница.
