# IvkTypeResolver Unification Plan

## Проблема

Определение типа ИВК (`TN01`/`TN02`) реализовано раздельно в backend и configurator frontend, с разными правилами.

- Backend (`tn.docgeneral`) определяет тип по активной строке подключения (`DBConnectionStrings.Where(x => x.Use)`), и сейчас использует жесткий `switch` по имени БД.
- Frontend (`configurator`) определяет тип по всем строкам подключения и отдельным правилам для БД/OPC.

Из-за этого одно и то же устройство может получать разный тип ИВК в разных частях системы.

## Текущее влияние

- Риск неправильного выбора режима логики документов (в т.ч. отображения направлений).
- Риск ложного состояния "Не выбрано" в конфигураторе при валидной backend-конфигурации.
- Рост вероятности регрессий: любое изменение правил нужно синхронно править в двух местах.

## Цель

Сделать единый источник истины для определения типа ИВК и применения пресетов, чтобы backend и frontend работали по одинаковым правилам.

## Путь решения

### Шаг 1. Зафиксировать единую спецификацию правил

Создать таблицу правил (алиасы БД + сопоставление с типом ИВК), например:

- `IVK_TN_01` -> `TN01`
- `ivk_tn`, `IVK_TN_02*` -> `TN02`

Примечание: финальный список алиасов должен быть согласован с реальными объектами/конфигами.

### Шаг 2. Внедрить единый resolver в backend

- Вынести логику в отдельный `IvkTypeResolver` (или аналогичный сервис).
- Перевести `GetDeviceType(...)` на этот resolver.
- Явно зафиксировать fallback-поведение и уровень логирования для неизвестных имен БД.

### Шаг 3. Синхронизировать frontend

- Перевести `detectIvkType/applyIvkType` на ту же спецификацию.
- Учитывать только активные подключения (`Use == true`) или формально описать другое правило и применять его в обеих частях одинаково.

### Шаг 4. Единые тест-кейсы

Сформировать единый набор сценариев и проверить его и в backend, и во frontend:

- `IVK_TN_01` -> `TN01`
- `ivk_tn` -> `TN02`
- `IVK_TN_02_test` -> `TN02`
- неизвестное имя -> fallback (должен быть одинаковый)
- пустые/отключенные строки подключения

## Критерии готовности

- Backend и frontend возвращают одинаковый тип ИВК для одинаковых входных данных.
- Добавление нового алиаса делается в одном месте спецификации и покрывается тестом.
- Отсутствуют противоречия между unit-тестами backend и frontend.

## Ожидаемый результат

Снижение риска регрессий при работе с типами ИВК, предсказуемое поведение конфигуратора и docgeneral, упрощение сопровождения.
