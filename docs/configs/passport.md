## Паспорт качества: конфигурации

### Текущая структура конфигурации

Актуальные конфигурационные файлы паспортов находятся в `TN_Doc/Cfg/Passport/`:
- `CfgEditPassport_GOSTR50.2.040(I).json`
- `CfgEditPassport_GOSTR50.2.040(G).json`
- `CfgEditPassport_MI3532(13).json`
- `CfgEditPassport_MI3532(14).json`
- `CfgEditPassport_MI3532(15).json`
- `CfgEditPassport_MI3532(15)_China.json`
- `CfgEditPassportExport.json`
- `CfgEditPassportForActNP.json`
- `CfgEditPassport_EAC.json`

Каждый файл содержит разделы:
- `Methods[]` - методы испытаний для параметров
- `Parameters[]` - параметры качества с полями:
  - `Id` - идентификатор параметра
  - `Key` - ключ параметра (например, `"TempCorrection"`)
  - `Name` - отображаемое название
  - `Use` - использовать параметр
  - `Edit` - разрешить редактирование

### Матрица устройств и активация ELIS

⚠️ **ВНИМАНИЕ:** Интеграция с ELIS в текущей версии реализована частично:
- Присутствуют классы данных ELIS (`QualityPassport`, `ParameterInfo`) в субмодуле `tn.docgeneral`
- Есть контроллер `ElisController` для обработки ошибок ELIS
- НО: отсутствует Document Editor и клиентская часть для загрузки данных из ELIS

Конфигурация ELIS должна находиться в `TN_Doc/Cfg/CfgApp.json`, но из-за большого размера файла требуется проверка вручную.

### Поле `Edit` для параметров качества

**Назначение:** определяет, может ли оператор редактировать значение параметра в текущем HTML‑редакторе.

**Значения:**
- `true` — параметр доступен для редактирования
- `false` — параметр отображается как read‑only (значение подставляется из данных, без возможности редактирования)

**Примечание:** Логика обработки `Edit` реализована в `tn.docgeneral/Passport/DocPassport.cs` при построении HTML‑формы.

### Поле `IsDefault` для методов испытаний

✅ **СТАТУС:** Поле `IsDefault` реализовано и используется во всех конфигурационных файлах методов.

**Назначение:** Определяет, должен ли метод выбираться автоматически при заполнении паспорта качества.

**Значения:**
- `true` — метод автоматически подставляется при создании/редактировании документа (только один метод должен иметь `IsDefault = true` для каждого параметра)
- `false` — метод доступен для выбора, но не выбирается автоматически

**Пример конфигурации:**
```json
{
  "Methods": [
    {
      "Id": 1,
      "Name": "ГОСТ 2477-2014 метод А",
      "LimitValue": 0,
      "LimitValueString": "-",
      "IsDefault": true
    },
    {
      "Id": 2,
      "Name": "ГОСТ 2477-2014 метод Б",
      "LimitValue": 0,
      "LimitValueString": "-",
      "IsDefault": false
    }
  ]
}
```

**Текущая реализация:**
- При открытии редактора документа метод с `IsDefault = true` автоматически подставляется как выбранный
- В каждом массиве методов для параметра только один метод должен иметь `IsDefault = true`
- При отсутствии метода с `IsDefault = true` требуется явный выбор оператора
- Незаполненные методы отображаются с placeholder "Метод не выбран" для явной индикации

**Использование в Document Editor:**
- Методы с `IsDefault = false` не выбираются автоматически — требуется явный выбор оператора
- При отсутствии выбранного метода показывается placeholder "Метод не выбран"

### Порядок обновления конфигов

1. **Инвентаризация**
   - Сформировать список файлов через `glob TN_Doc/Cfg/**/CfgEditPassport*.json`.
   - Зафиксировать новые конфиги/устройства в `passport_devices_useElis.json`.

2. **Правки эталона**
   - Внести изменения в `TN_Doc/Cfg/Passport/CfgEditPassport_GOSTR50.2.040(I).json`.
   - Проверить валидность: `jq '.' TN_Doc/Cfg/Passport/CfgEditPassport_GOSTR50.2.040(I).json`.

3. **Тиражирование**
   - Синхронизировать остальные `CfgEditPassport*.json` (включая `Cfg/Passport` и корневые `Cfg/`).
   - Для файлов с наследованными комментариями `/* ... */` допустимо временно их удалять при автоматической обработке, но комментарий нужно вернуть в исходное место.

4. **Проверки**
   - Минимум три файла прогнать через `jq '.'` (пример: MI3532(15), EAC, Export).
   - Запустить `dotnet test Tests/Tests.csproj --filter CfgEditPassport` (см. раздел о тестах).
   - Запустить `dotnet test Tests/Tests.Unit --filter "ClassName=ConfigEncodingTests"` — проверка UTF-8, U+FFFD и валидности JSON для всех `TN_Doc/Cfg`.
   - Ручной smoke: открыть паспорт на устройстве с `UseElis=true` и на устройстве с `UseElis=false`, убедиться что редактор грузится без ошибок.

5. **Согласование**
   - Сохранить диффы и отправить эксплуатационщикам ответственных площадок (ГОСТ/МИ/ЕАЭС) перед выкладкой.

### Rollback

- Конфиги лежат в git, поэтому откат выполняется командой `git checkout -- <path>` + публикация через обычный процесс доставки.
- При необходимости откатить только одно устройство:
  1. `git show <commit> -- TN_Doc/Cfg/Passport/CfgEditPassport_<variant>.json > backup.json`.
  2. Скопировать файл на стенд и перезапустить сервисы TN_Doc.
- После отката повторить smoke на устройствах `UseElis=true/false` и обновить `passport_devices_useElis.json`, чтобы отразить фактическое состояние.

### Тесты

- Основные проверки конфигурации паспортов сосредоточены в `Tests/Configs/CfgEditPassportTests.cs`
- Команда прогона в репозитории:
  ```bash
  dotnet test Tests/Tests.csproj --filter CfgEditPassport
  ```
- Перед сдачей изменений фиксируйте, какие файлы проходили валидацию `jq`, и прикладывайте результаты тестов в описании merge request.
