## Паспорт качества: конфигурации

> **ВАЖНО:** Описанные в этом документе поля конфигурации (`IsBallast`, `SlaveKey`, `LinkedParameter`, `IsDefault`) **НЕ ПРИСУТСТВУЮТ** в текущих конфигурационных файлах `TN_Doc/Cfg/Passport/CfgEditPassport*.json`. Они описаны как планируемая функциональность.

### Текущая структура конфигурации

Актуальные конфигурационные файлы паспортов находятся в `TN_Doc/Cfg/Passport/`:
- `CfgEditPassport_GOSTR50.2.040(I).json`
- `CfgEditPassport_GOSTR50.2.040(G).json`

- `CfgEditPassportExport.json`
- `CfgEditPassportForActNP.json`
- `CfgEditPassport_EAC.json`

Каждый файл содержит разделы:
- `Methods[]` - методы испытаний для параметров
- `Parameters[]` - параметры качества с полями:
  - `Id` - идентификатор параметра
  - `Key` - ключ параметра (например, `"TempCorrection"`)
  - `Name` - отображаемое название
  - `Use` - использовать параметр
  - `Edit` - разрешить редактирование

### Поле `IsBallast` (ПЛАНИРУЕТСЯ)

⚠️ **СТАТУС:** Поле `IsBallast` **НЕ ПРИСУТСТВУЕТ** в текущих конфигурационных файлах.

**Планируемое назначение:**
- `true` — показатель считается балластным, столбец Result синхронизируется с Measurement
- `false` — поведение прежнее, Result редактируется операторами
- Отсутствие поля трактуется как `false` (значение по умолчанию в модели)

**Планируемая классификация:**
- **Балластные**: `TempCorrection`, `PressCorrection`, `DensCorrection`, `Dens20Correction`, `Dens15Correction`, `MassWaterFracCorrection`, `Chloride_Salts.Concentration`, `Chloride_Salts.MassFraction`, `Impurity`
- **Небалластные**: `SulfurCorrection`, `DNP.kPa`, `DNP.mercury_mm`, `Yield_fraction_200`, `Yield_fraction_300`, `Yield_fraction_350`, `Mass_fraction_of_paraffin`, `Mass_fraction_of_hydrogen_sulfide`, `Mass_fraction_of_methyl_and_ethyl_mercaptan`, `Mass_fraction_of_organic_chlorides`

**Планируемое использование в Document Editor:**
- Балластные параметры (`IsBallast = true`) не будут позволять ручное редактирование результата
- Кнопка редактирования результата будет скрываться для балластных параметров
- При изменении измерения балластного параметра результат будет автоматически обновляться
- Небалластные параметры (`IsBallast = false`) будут иметь кнопку "Редактировать" для ручной корректировки результата

### Связанные параметры (Master-Slave) (ПЛАНИРУЕТСЯ)

⚠️ **СТАТУС:** Поле `SlaveKey` **НЕ ПРИСУТСТВУЕТ** в текущих конфигурационных файлах.

**Планируемое назначение:** Определение связи между параметрами, которые представляют одну и ту же физическую величину в разных единицах измерения.

**Планируемый механизм:** Использование поля `SlaveKey` в конфигурации параметра для указания на связанный (ведомый) параметр.

**Правила:**
- Если у параметра есть поле `SlaveKey` и оно ссылается на существующий параметр → этот параметр является **мастером (master)**
- Параметр, на который ссылается `SlaveKey` → является **слейвом (slave)**
- Если поле `SlaveKey` отсутствует → параметр обрабатывается независимо

**Примеры связанных пар:**

1. **Давление насыщенных паров:**
   - Мастер: `DNP.kPa` (кПа) → Слейв: `DNP.mercury_mm` (мм рт. ст.)
   - Коэффициент пересчета: 1 кПа = 7.50062 мм рт. ст.

2. **Хлористые соли:**
   - Мастер: `Chloride_Salts.Concentration` (мг/дм³) → Слейв: `Chloride_Salts.MassFraction` (%)
   - Пересчет зависит от плотности нефти

**Пример конфигурации:**
```json
{
  "Id": 11,
  "Key": "DNP.kPa",
  "Name": "Давление насыщенных паров, кПа",
  "SlaveKey": "DNP.mercury_mm",
  "RequiredFill": true,
  "RoundValue": 1,
  "Use": true,
  "Edit": true,
  "IsBallast": false
}
```

**Планируемое поведение при сохранении:**
- При сохранении паспорта качества флаг `Legal = 0` будет устанавливаться для:
  - **Slave-параметров** — значение вычисляется от Master-параметра
  - **Нередактируемых параметров** (`Edit = false`) — значение приходит из ИВК, оператор не может его изменить
- Флаг `Legal = 0` будет означать, что параметр не участвует в легализации документа
- Только редактируемые Master/Independent параметры будут учитываться при валидации

**Планируемая реализация:**
- Связь между параметрами будет зафиксирована в конфигурации через поле `SlaveKey`
- Slave-параметры будут полностью скрыты из таблицы показателей качества в Document Editor
- Slave-параметры не будут участвовать в валидации формы (не блокируют кнопку "Сохранить")
- Значения Slave-параметров будут вычисляться автоматически в ИВК от Master-параметра
- Логика автоматического пересчета значений будет реализована в будущих версиях

**Целевые файлы конфигурации:**
- `TN_Doc/Cfg/Passport/CfgEditPassport_GOSTR50.2.040(I).json`
- `TN_Doc/Cfg/Passport/CfgEditPassport_GOSTR50.2.040(G).json`

### Связанные параметры с общим методом (LinkedParameters) (ПЛАНИРУЕТСЯ)

⚠️ **СТАТУС:** Поле `LinkedParameter` **НЕ ПРИСУТСТВУЕТ** в текущих конфигурационных файлах.

**Планируемое назначение:** Объединение выбора метода испытаний для связанных параметров качества, когда между ними **не** установлена Slave-связь.

**Контекст:** В таблице показателей качества паспорта существуют пары параметров, представляющих одну и ту же величину в разных единицах измерения. Для таких пар логично использовать один и тот же метод испытаний.

**Планируемое отличие от SlaveKey:**
| Механизм | SlaveKey | LinkedParameter |
|----------|----------|-----------------|
| Видимость | Slave-параметр скрыт | Оба параметра видимы |
| Редактирование | Только Master | Оба параметра редактируемы |
| Комбобокс метода | Один (у Master) | Один объединённый |
| Область применения | ГОСТ Р 50.2.040 | МИ 3532 и другие |

**Планируемый механизм:** Использование поля `LinkedParameter` в конфигурации ведущего параметра для указания ведомого.

**Правила:**
1. Если у параметра есть `LinkedParameter` → параметр является **ведущим (leader)**
2. Параметр, на который ссылается `LinkedParameter` → является **ведомым (follower)**
3. `SlaveKey` имеет приоритет: если указаны оба поля, `LinkedParameter` игнорируется
4. При выборе метода для ведущего параметра — метод автоматически записывается в оба параметра

**Пары параметров с LinkedParameter:**

| Ведущий параметр | Ведомый параметр |
|------------------|------------------|
| `Chloride_Salts.Concentration` (мг/дм³) | `Chloride_Salts.MassFraction` (%) |
| `DNP.kPa` (кПа) | `DNP.mercury_mm` (мм рт. ст.) |
| `Yield_fraction_200` (до 200°С) | `Yield_fraction_300` (до 300°С) |

**Пример конфигурации:**
```json
{
  "Id": 7,
  "Key": "Chloride_Salts.Concentration",
  "Name": "Массовая концентрация хлористых солей, мг/дм³",
  "LinkedParameter": "Chloride_Salts.MassFraction",
  "Use": true,
  "Edit": true,
  "IsBallast": true
}
```

**Планируемое поведение в UI (Document Editor):**
- Комбобокс метода будет отображаться с `rowspan` объединяя строки связанных параметров
- Колонка "Документ" также будет объединяться (для ELIS интеграции)
- Ведомый параметр не будет иметь собственного комбобокса метода
- Оба параметра будут сохранять независимое редактирование Измерения и Результата

**Планируемая визуализация:**
```
| № | Наименование                        | Метод      | Измер. | Результат |
|----|-------------------------------------|------------|--------|-----------|
| 7  | Хлористые соли, мг/дм³              | [Общий     | 100    | 100       |
| 8  | Хлористые соли, %                   |  комбо]    | 0.01   | 0.01      |
```

**Планируемое поведение при сохранении (бэкенд):**
- При сохранении метода ведущего параметра → метод будет копироваться в ведомый
- Оба параметра будут получать одинаковое значение метода в базе данных
- `Legal` флаги будут устанавливаться независимо для каждого параметра

**Целевые файлы конфигурации:**
- `TN_Doc/Cfg/Passport/CfgEditPassport_MI3532(13).json`
- `TN_Doc/Cfg/Passport/CfgEditPassport_MI3532(14).json`
- `TN_Doc/Cfg/Passport/CfgEditPassport_MI3532(15).json`
- `TN_Doc/Cfg/Passport/CfgEditPassport_MI3532(15)_China.json`

### Поле `Edit` для параметров качества
**Назначение:** определяет, может ли оператор редактировать значение параметра в текущем HTML‑редакторе.

### Поле `IsDefault` для методов испытаний (ПЛАНИРУЕТСЯ)

⚠️ **СТАТУС:** Поле `IsDefault` **НЕ ПРИСУТСТВУЕТ** в текущих конфигурационных файлах методов.

**Планируемое назначение:** Определение, должен ли метод выбираться автоматически при заполнении паспорта качества

**Планируемые значения:**
- `true` — метод будет выбираться автоматически при создании/редактировании документа (только один метод должен иметь `IsDefault = true` для параметра)
- `false` — метод доступен для выбора, но не выбирается автоматически

**Планируемая реализация:**
- В конфигурации методов можно будет указать `IsDefault = true` для метода по умолчанию
- При отсутствии метода с `IsDefault = true` будет требоваться явный выбор оператора
- Незаполненные методы будут отображаться с placeholder "Метод не выбран" для явной индикации

**Планируемое использование в Document Editor:**
- Методы с `IsDefault = false` не будут выбираться автоматически — потребуется явный выбор оператора
- При отсутствии выбранного метода будет показываться placeholder "Метод не выбран"
- Будет реализована визуальная индикация методов вне справочника

### Индикация методов вне справочника (ПЛАНИРУЕТСЯ)

⚠️ **СТАТУС:** Функционал **НЕ РЕАЛИЗОВАН** в текущей версии.

**Планируемое назначение:** Визуальное предупреждение для методов испытаний, которые отсутствуют в локальном справочнике конфигурации.

**Планируемые визуальные характеристики:**
- Желтая рамка вокруг поля выбора метода (цвет: `#f5c24c`)
- Текстовое предупреждение под полем: "отсутствует в справочнике"
- Будет применяться к методам, введенным вручную или полученным из ELIS, но не зарегистрированным в `CfgEditPassport.json`

**Планируемые технические детали:**
- Новое поле в типе `ParameterMethod`: `isInDictionary: boolean`
- Проверка наличия метода: регистронезависимое сравнение с `paramSchema.methodOptions`
- Компонент: `PassportMethodSelect.vue` с CSS классом `.unknown-method`

**Планируемое использование:**
- Будет помогать операторам идентифицировать нестандартные методы
- Не будет блокировать сохранение документа
- Будет требовать внимания для актуализации справочника `CfgEditPassport.json`

**Пример:** Если оператор введет метод "ГОСТ 12345-2023" вручную, но этот метод не зарегистрирован в конфигурации, поле будет подсвечено желтым с предупреждением

### Определение результата измерения (ResolveResultValue) (ПЛАНИРУЕТСЯ)

⚠️ **СТАТУС:** Алгоритм **НЕ РЕАЛИЗОВАН** в текущей версии. Файл `PassportQualityStrategies.cs` не найден в субмодуле.

**Планируемое назначение:** Алгоритм определения значения результата измерения для отображения в Document Editor.

**Планируемая логика:**

1. **Паспорт не заполнен (IsFilled == 0):**
   - Будет браться `CorrectionValue` (значение коррекции)
   - Fallback на `RawValue` если `CorrectionValue` пустой

2. **Паспорт заполнен (IsFilled > 0):**
   - Будет проверяться `LabInfo.Value` на соответствие паттерну "слово число"
   - Если соответствует — будет возвращаться `LabInfo.Value`
   - Иначе — будет возвращаться `ResultValue`

**Планируемый паттерн "слово число":**
- Регулярное выражение: `^[а-яА-ЯёЁa-zA-Z]+\s+\d+[.,]\d+$`
- Примеры совпадений:
  - `Менее 4.0`
  - `более 5,02`
  - `Less 0,5`
- Примеры несовпадений:
  - `4.0` (нет слова)
  - `Менее` (нет числа)
  - `< 4.0` (спецсимвол вместо слова)

**Планируемый файл реализации:** `tn.docgeneral/Passport/PassportQualityStrategies.cs`

### Порядок обновления конфигов
1. **Инвентаризация**  
   - Сформировать список файлов через `glob TN_Doc/Cfg/**/CfgEditPassport*.json`.  
   - Зафиксировать новые конфиги/устройства в `passport_devices_useElis.json`.
2. **Правки эталона**  
   - Внести изменения в `TN_Doc/Cfg/Passport/CfgEditPassport_GOSTR50.2.040(I).json`.  
   - Проверить валидность: `jq '.' TN_Doc/Cfg/Passport/CfgEditPassport_GOSTR50.2.040(I).json`.
3. **Тиражирование**  
   - Синхронизировать остальные `CfgEditPassport*.json` (включая `Cfg/Passport` и корневые `Cfg/`).  
   - Для файлов с наследованными комментариями `/* ... */` допустимо временно их удалять при автоматической обработке, но комментарий нужно вернуть в исходное место.
4. **Проверки**  
   - Минимум три файла прогнать через `jq '.'` (пример: MI3532(15), EAC, Export).  
   - Запустить `dotnet test Tests/Tests.csproj --filter CfgEditPassport` (см. раздел о тестах).  
   - Ручной smoke: открыть паспорт на устройстве с `UseElis=true` и на устройстве с `UseElis=false`, убедиться что редактор грузится без ошибок.
5. **Согласование**  
   - Сохранить диффы и отправить эксплуатационщикам ответственных площадок (ГОСТ/МИ/ЕАЭС) перед выкладкой.

### Rollback
- Конфиги лежат в git, поэтому откат выполняется командой `git checkout -- <path>` + публикация через обычный процесс доставки.  
- При необходимости откатить только одно устройство:
  1. `git show <commit> -- TN_Doc/Cfg/Passport/CfgEditPassport_<variant>.json > backup.json`.
  2. Скопировать файл на стенд и перезапустить сервисы TN_Doc.
- После отката повторить smoke на устройствах `UseElis=true/false` и обновить `passport_devices_useElis.json`, чтобы отразить фактическое состояние.
